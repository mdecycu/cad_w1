Git

循序設計流程:

由組長透過 IPv6 位址啟動分組動態網站，逐一由組員登入維護網站內容，階段編輯結束後轉為靜態，再由組長進行提交推送。

優點:

同步協同設計流程:

所有組員同時在負責維護的頁面，自行建立分支進行維護與推送後，透過拉回請求讓 reviewer 檢查後合併進入 main 分支，並轉為靜態網站。

優點:

各組員可以同步進行網站內容修改，且透過提交訊息可清楚了解每一位組員在不同設計階段的貢獻

缺點:

流程比循序設計複雜許多，各組員必須清楚如何使用 git 分支、合併、拉回請求等指令的使用時機。

由於練習用的 https://github.com/mdecycu/cad_w1 倉儲並沒有將組員內建為協同者，因此組長以 https://github.com/mdecycu/cad2025_hw 作為 Template 建立練習用的協同倉儲後，必須在倉儲的 Settings - Collaborators 將組員納入作為倉儲的協同者。

假如利用 Codespaces 管理網站，需要先執行 ./init_replit 安裝啟動所需模組，之後利用 python main.py& 在背景執行動態網站，以 python main2.py& 在背景執行靜態網站。

其中動態網站用來編輯內容，編輯完成後按下 Convert 就可以轉為靜態網站內容，並在在 python main2.py& 所啟動的瀏覽器頁面中檢視。

w1_command.txt

各組員一開始分別負責各頁面內容時，採建立對應分支後進行編輯，轉為靜態網站確定內容後進行提交至該分支，然後建立 Pull Request 指令由組長或其他組員進行查驗後與 main 分支進行合併。

負責其他頁面內容編輯的組員，則可在進行編輯之前儘量在最新 main 分支的版本中，建立對應分支後進行編輯，其後轉為靜態網站內容後，再利用 Pull Request 指定 reviewer 進行查驗與後續的 main 分支進行合併。
